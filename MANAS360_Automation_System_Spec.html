1 Standardized Webhook Payload

ONE consistent format used by ALL events. Backend sends this to Zoho Flow's webhook URL.

Universal Payload Structure

JSON 
{
  "event": "USER_REGISTERED",        // REQUIRED — event type (uppercase, underscore-separated)
  "role": "PATIENT",                 // REQUIRED — PATIENT | THERAPIST | PSYCHIATRIST | COACH
  "timestamp": "2026-04-06T12:30:00Z", // REQUIRED — ISO 8601
  "source": "MANAS360",               // REQUIRED — always "MANAS360"
  "data": {
    "userId": "u_1001",              // REQUIRED — unique user ID
    "name": "Dr. Priya Sharma",       // REQUIRED — display name
    "phone": "919636269114",          // REQUIRED — WATI format: country code, no +, no spaces
    "email": "priya@example.com",     // REQUIRED — for email actions
    "therapistName": "Dr. Rajesh",    // OPTIONAL — for session events
    "date": "2026-04-10",             // OPTIONAL — for session events (YYYY-MM-DD)
    "time": "6:00 PM IST",            // OPTIONAL — for session events
    "meetingLink": "https://...",      // OPTIONAL — Jitsi URL for sessions
    "detailsLink": "https://...",      // OPTIONAL — deep link to platform
    "certificateName": "CBT Level 1",  // OPTIONAL — for certificate events
    "amount": "₹699",                 // OPTIONAL — for payment events
    "status": "success",              // OPTIONAL — for payment events
    "specialization": "Clinical Psychologist", // OPTIONAL — for provider registration
    "clinicalType": "PHQ9_CRITICAL"    // OPTIONAL — for clinical events
  }
}
Important contract note: this is the business event payload that the backend should emit to Zoho Flow for routing. The WhatsApp sender path is a separate payload shape used only when the backend is preparing a WATI message send. Do not mix the two schemas.

Naming Conventions

Rules (enforce in backend helper):
• event: UPPER_SNAKE_CASE — USER_REGISTERED, SESSION_BOOKED
• role: UPPER_SNAKE_CASE — PATIENT, THERAPIST, PSYCHIATRIST, COACH
• data.*: camelCase — userId, therapistName, meetingLink
• phone: Always 919636269114 format (country code, no +, no spaces, no dashes)
• timestamp: Always ISO 8601 UTC — 2026-04-06T12:30:00.000Z
• Never send null — omit the field entirely or send empty string "". Null values break Zoho Flow variable resolution.

Backend Helper (Node.js)

JavaScript 
// backend/src/helpers/zohoWebhook.js
const axios = require('axios');

const ZOHO_WEBHOOK_URL = process.env.ZOHO_FLOW_WEBHOOK_URL;

/**
 * Send event to Zoho Flow. Strips undefined/null fields.
 * Phone auto-formatted to WATI format (919636269114).
 */
async function emitEvent(event, role, data) {
  // Strip null/undefined from data
  const cleanData = {};
  Object.entries(data).forEach(([k, v]) => {
    if (v !== null && v !== undefined) cleanData[k] = v;
  });

  // Normalize phone: strip +, spaces, dashes
  if (cleanData.phone) {
    cleanData.phone = cleanData.phone.replace(/[\+\s\-\(\)]/g, '');
    // Ensure starts with country code
    if (cleanData.phone.length === 10) cleanData.phone = '91' + cleanData.phone;
  }

  const payload = {
    event: event.toUpperCase(),
    role: role.toUpperCase(),
    timestamp: new Date().toISOString(),
    source: 'MANAS360',
    data: cleanData,
  };

  try {
    await axios.post(ZOHO_WEBHOOK_URL, payload, {
      headers: { 'Content-Type': 'application/json' },
      timeout: 5000,
    });
    console.log(`[ZohoFlow] Emitted: ${event}/${role}`);
  } catch (err) {
    // Fire-and-forget — log but don't throw
    console.error(`[ZohoFlow] Failed: ${event}/${role}`, err.message);
  }
}

module.exports = { emitEvent };
Current backend note: the WhatsApp service in the codebase already emits a separate Zoho Flow payload for message sending with event: "whatsapp_message_send" and data.phoneNumber, data.templateName, data.templateType, data.userType, data.variables, and data.language. That is different from the business-event payload above, so keep them separate unless you intentionally refactor into one unified contract.

Why NOT Use Zoho's Native WATI Connector

Confirmed Bug: Zoho Flow's built-in WATI "Send Template Message" action strips the parameters array from the API request before sending it to WATI. WATI receives a template name but no variable values → returns "Something went wrong." This is documented in our WATI Fix Guide (April 2026).

Solution: Use Zoho Flow's generic "Webhooks → POST" action to call WATI's REST API directly. This gives full control over the request URL, headers, and body — Zoho can't strip what it doesn't manage.

Why Zoho Flow as Orchestration Layer

Zoho Flow is chosen because:
• Already in the locked Zoho ecosystem (Desk, Sign, ZeptoMail, Campaigns)
• Visual builder — non-developers can see and modify notification flows
• Decision blocks handle event/role routing without code
• Native connections to ZeptoMail, Zoho Desk, Zoho Campaigns
• Webhook trigger accepts any JSON payload
• Free tier supports the current event volume

What it replaces: Custom notification service in the backend. Backend stays focused on business logic; Zoho Flow handles "tell someone what happened."

4 Zoho Flow Design (Step-by-Step)

How to build the flow in Zoho Flow's visual builder. Chandu follows these exact steps.

Step 1: Create the Webhook Trigger

Zoho Flow Setup 
1. Go to Zoho Flow → Create Flow → "MANAS360 Event Router"
2. Trigger: "Webhook" → "Incoming Webhook"
3. Zoho generates a URL like:
   https://flow.zoho.com/webhook/catch/XXXXX/YYYYY/ZZZZZ

4. CRITICAL: First POST (schema capture)
   - Run ONE test curl (see Section 10) with ALL fields populated
   - Zoho captures the schema from this first POST
   - All subsequent variable paths depend on this schema

5. Copy the FULL webhook URL → set as ZOHO_FLOW_WEBHOOK_URL in backend .env
Schema Mismatch Prevention: The first POST to a new webhook defines the schema. If you send a payload missing fields (e.g., no therapistName), Zoho will NOT recognize that field in later POSTs. Always send the FIRST test with ALL possible fields populated (use placeholder values like "N/A" instead of omitting).

Step 2: Decision Block — Event Routing

Zoho Flow Config 
Add: Decision block after Webhook trigger
Field: ${webhookTrigger.payload.event}

Conditions:
  condition1: equals "USER_REGISTERED"
  condition2: equals "SESSION_BOOKED"
  condition3: equals "SESSION_FOLLOWUP"
  condition4: equals "PAYMENT_SUCCESS"
  condition5: equals "PAYMENT_FAILED"
  condition6: equals "CLINICAL_EVENT"
  condition7: equals "CERTIFICATE_STARTED"
  condition8: equals "CERTIFICATE_COMPLETED"
  Default:    → Log action ("Unknown event")

Step 3: Nested Decision — Role Routing

Zoho Flow Config 
Inside condition1 (USER_REGISTERED):
  Add: Decision block
  Field: ${webhookTrigger.payload.role}

  condition1: equals "PATIENT"
  condition2: equals "THERAPIST"
  condition3: equals "PSYCHIATRIST"
  condition4: equals "COACH"
  Default:    → Log action ("Unknown role")

Zoho Flow Config

    Authorization: Bearer {YOUR_WATI_TOKEN}

5 WATI API Configuration

Direct REST API calls from Zoho Flow's Webhooks POST action. Bypasses the broken native connector.

Endpoint

WATI API v2 
POST https://live-mt-server.wati.io/{TENANT_ID}/api/v2/sendTemplateMessage?whatsappNumber={PHONE}

// TENANT_ID: Found in WATI dashboard → API Docs page
// PHONE: Query parameter — format 919636269114 (no +)

Headers

HTTP Headers 
Authorization: Bearer {YOUR_WATI_API_TOKEN}
Content-Type: application/json

Body Structure

JSON 
{
  "template_name": "user_welcome_patient",
  "broadcast_name": "zoho_flow_patient_welcome",
  "parameters": [
    { "name": "1", "value": "Priya" },
    { "name": "2", "value": "MANAS360" }
  ]
}

Parameter Rules

Critical WATI Parameter Rules:
• Parameters use numbered names: "1", "2", "3" — matching {{1}}, {{2}}, {{3}} in the template
• Parameter order matters — they must match the template's variable order exactly
• Count must match — if template has 3 variables, send exactly 3 parameters
• broadcast_name is required — use a unique identifier per flow for tracking
• Phone in URL query param — no + sign, include country code: 919636269114

WATI API Token Generation

Steps 
1. Log into WATI → API (left nav) OR Connector → API → Create API Token
2. Click "Generate new token"
3. Name: "zoho-flow-production"
4. Scopes: contacts:read, contacts:write, messagetemplate:read
5. Copy token → store in Zoho Flow's Webhooks POST header
6. Store securely — do NOT commit to Git

Debugging Checklist

Debug Steps 
1. curl returns HTTP 200? → Zoho accepted the POST
   If 404: Wrong webhook URL (check FLOW_ID in URL matches builder)
   If 410: Flow deleted/unpublished — copy fresh webhook URL

2. Check Zoho Flow → History tab
   → Find the run with matching timestamp
   → Click to expand each block
   → Green = passed, Red = failed

3. If Decision block shows "Default":
   → Variable path mismatch
   → Check: ${webhookTrigger.payload.event} (not ${webhookTrigger.event})
   → The "payload" layer exists because Zoho wraps webhook data

4. If Webhooks POST to WATI fails:
   → Click the block → inspect the response body
   → "Something went wrong" = parameters stripped (use Webhooks POST, not native WATI)
   → "Invalid template" = template_name doesn't match WATI dashboard
   → "Parameter mismatch" = wrong number of parameters
   → 401 = Bearer token expired → regenerate in WATI

5. If email fails:
   → Check ZeptoMail connection status in Zoho Flow → Connections
   → Verify "from" address is validated in ZeptoMail dashboard

11 Common Errors + Fixes

ErrorCauseFix
Null parameters in WATI Backend sent null for a field → Zoho resolves ${data.therapistName} as empty → WATI rejects Never send null. Use "" or omit the field. The backend emitEvent() helper strips nulls.
Invalid phone format WATI requires 919636269114 but received +91 9636 269114 Backend helper strips all non-numeric chars and prepends 91 if 10-digit number.
Webhook schema mismatch First POST was missing fields → Zoho doesn't recognize later fields Delete the webhook trigger → recreate → send first POST with ALL fields populated (empty strings OK).
Wrong variable path Using ${webhookTrigger.data.name} instead of ${webhookTrigger.payload.data.name} Always include the payload layer: ${webhookTrigger.payload.data.name}
Template mismatch Template name in Zoho body doesn't match WATI dashboard Copy the exact element_name from WATI → Message Templates (not the display name).
Debug vs Live mode Flow is in "Draft" mode → webhook only works in Test & Debug, not on live POST Click "Publish" in Zoho Flow builder. Draft flows only run in manual test mode.
WATI "Something went wrong" Using native Zoho WATI connector (strips parameters) Replace with Webhooks POST action calling WATI API directly. See WATI Fix Guide.
OAuth token expired WATI Bearer token has an expiry date Generate a new token in WATI → API Tokens. Set no expiry for production tokens.
Flow runs but no WhatsApp Template not approved by Meta yet Check WATI → Message Templates → status must be "Approved". Pending/Rejected templates won't send.
Duplicate messages Backend retries on timeout → Zoho receives same event twice Make emitEvent() fire-and-forget (no retry). Log failures but don't re-send. Add idempotencyKey to payload if needed.

12 Production Best Practices

1. Centralized Payload Strategy

All events use the same emitEvent(event, role, data) helper. One function, one format, one webhook URL. Never build event-specific webhook endpoints — Zoho Flow handles routing via Decision blocks.

2. Reusable Template Design

WATI templates share parameter patterns: {{1}} is always the user's name, {{2}} varies by context. This means the first parameter mapping in Zoho Flow is always "1" → data.name. Consistency reduces errors.

3. Retry Mechanism

Strategy 
Backend → Zoho: Fire-and-forget. No retry. Log failures.
Zoho → WATI:    Zoho Flow has built-in retry (1 retry on failure).
Zoho → Email:   ZeptoMail is reliable (99%+ delivery). No custom retry needed.

If WATI fails after Zoho's retry:
  → Zoho Flow logs the failure in History
  → Configure Zoho Flow alerts (Settings → Notifications → On Failure)
  → Admin receives email about failed flow runs

4. Logging Strategy

Strategy 
Backend:
  → console.log(`[ZohoFlow] Emitted: ${event}/${role} for user ${data.userId}`)
  → console.error(`[ZohoFlow] Failed: ${event}/${role}`, err.message)
  → Optional: Write to mdc_audit_log table

Zoho Flow:
  → History tab shows every run with input/output per block
  → Enable email notifications for failed runs
  → Export History data monthly for audit

WATI:
  → Use broadcast_name per flow for tracking
  → WATI dashboard → Broadcast Reports → filter by broadcast_name
  → Track delivery/read/reply rates

5. Scalability Design

Strategy 
Current:
  1 Zoho Flow with Decision blocks handles all events
  Good for: 0–1,000 events/day

Scale trigger (1,000+ events/day):
  Split into multiple flows:
  → Flow 1: User Registration events
  → Flow 2: Session events
  → Flow 3: Payment events
  → Flow 4: Clinical + Certificate events
  Each flow has its own webhook URL
  Backend emitEvent() routes to correct URL based on event type

Scale trigger (10,000+ events/day):
  → Consider Bull queue in backend to buffer events
  → Queue drains to Zoho at controlled rate
  → Prevents Zoho Flow throttling

6. Security Checklist

Production Security:
☐ WATI API token stored in Zoho Flow, NOT in Git
☐ Zoho webhook URL stored in backend .env, NOT hardcoded
☐ No patient PII in Zoho Desk ticket bodies (use IDs, not full details)
☐ WATI templates don't include sensitive clinical data
☐ CLINICAL_EVENT alerts go to therapist's phone, NOT the patient's
☐ Admin email alerts use admin@manas360.com, NOT personal emails
☐ Zoho Flow History auto-purges after 30 days (check retention settings)
